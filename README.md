# Wireshark AI — SOC Traffic Analysis Assistant

Captures or reads network traffic, learns a baseline, detects anomalies, scores each alert with a confidence level, and answers plain-language analyst questions — without manually reading every packet.

---

## SOC Use Cases

This system is designed to support real Security Operations Center (SOC) workflows. It combines **rule-based detection**, **ML anomaly scoring**, and **multi-stage correlation** — the same hybrid approach used in production SIEM platforms.

| Scenario | Detection Method | MITRE ATT&CK |
|----------|-----------------|--------------|
| SYN Flood / DoS attack | High SYN ratio + many destination IPs | T1498.001 |
| Port scanning activity | Many destination ports + short-lived flows | T1046 |
| Suspicious service access (SSH/RDP/SMB) | Connections to critical ports 22, 3389, 445 | T1021 |
| ARP cache poisoning / MiTM | Gratuitous ARP flood, MAC claiming multiple IPs | T1557.002 |
| Lateral movement / host sweep | Fan-out to many hosts with few packets per flow | T1021 |
| ARP host discovery sweep | Automated ARP scan across many targets | T1018 |
| ICS/SCADA HMI reconnaissance | HTTP access to industrial control endpoints | T1071.001 |
| SSH brute force | High SYN rate to SSH, all connections rejected | T1110 |
| SSH tunneling | SSH session on non-standard port | T1572 |
| Cleartext credential exposure | Telnet protocol detected | T1040 |
| Distributed SYN flood (DDoS) | Many source IPs all sending SYN | T1498.001 |
| Web path enumeration | Many unique HTTP URIs in one window | T1083 |

## Detection Architecture

```
Packet stream
     │
     ▼
Rule-based engine ──► Immediate flag for known attack signatures
     │                (ARP poisoning, ICS recon, SYN flood, port scan)
     ▼
ML anomaly scorer ──► IsolationForest trained on baseline traffic
     │                (catches unknown patterns the rules miss)
     ▼
Confidence engine ──► Scores 0–100% based on evidence strength,
     │                baseline deviation, source reputation,
     │                asset criticality, and flow direction
     ▼
Kill chain tracker ──► Correlates events per source IP across windows
     │                 (9 multi-stage intrusion patterns)
     ▼
SIEM-style alert (HIGH / MEDIUM / LOW) + Confidence % + MITRE ATT&CK mapping
```

> The LLM assistant (optional) does not make detection decisions. It only summarizes existing evidence generated by the rule-based engine, ML scorer, and triage module.

## Alert Output Format

Every suspicious detection prints a structured SOC alert block:

```
════════════════════════════════════════════════════════
[2026-05-04 21:04:32] ALERT — HIGH
────────────────────────────────────────────────────────
Threat   : Port Scan: 42 destination ports probed  [T1046]
Source   : HOST_A
Severity : HIGH
Confidence: 86%  |  FP Risk: LOW
Correlation: HOST_A has triggered 4 suspicious windows — persistent threat
Evidence :
  - Destination ports: 42
  - SYN ratio: 0.78
  - Packets: 342
Baseline :
  - Unique dest ports: 8.7x above baseline
  - Packet rate: 3.4x above baseline

Top Flows:
  1. HOST_A → HOST_B  [TCP, 120 pkts]
     Filter: ip.src == HOST_A && ip.dst == HOST_B
════════════════════════════════════════════════════════
```

Normal windows produce a single quiet line:

```
[21:04:42] NORMAL | Packets=87 | Flows=12
```

If the same alert fires repeatedly, it is **suppressed** after the first print:

```
[i] Suppressed 14 duplicate alert(s) from the same source in the last 10 minutes
```

If a multi-stage intrusion pattern is detected, a kill chain warning fires before the alert block:

```
[!] Kill Chain Pattern: Port scan → SSH brute force — possible credential compromise in progress
```

## Confidence Score and False Positive Risk

Each alert includes a **confidence score (0–100%)** and a **false positive risk (LOW / MEDIUM / HIGH)** computed from:

- Severity base (HIGH=55, MEDIUM=40, LOW=25)
- Evidence signals: SYN ratio, unique dst ports, avg pkts/flow, ARP indicators
- Baseline deviation (how far above the learned mean)
- Alert correlation (repeated alerts from the same source)
- Asset criticality (critical asset destination raises confidence)
- Source reputation (authorized scanner lowers confidence)
- Flow direction (noise traffic lowers confidence)

FP Risk is the inverse: confidence ≥ 70% → LOW, 45–69% → MEDIUM, < 45% → HIGH.

---

## Kill Chain Timeline

Events are tracked per source IP across windows. If a source triggers multiple suspicious events that match a known multi-stage pattern, an intrusion warning is raised.

**Recognized patterns:**

| Pattern | Warning |
|---------|---------|
| Port Scan → SSH Brute Force | Possible credential compromise in progress |
| ARP Cache Poisoning + Cleartext Protocol | Active MiTM interception detected |
| Port Scan → Lateral Movement | Post-compromise host pivot |
| Port Scan → Lateral Sweep | Multi-stage reconnaissance and pivot |
| ARP Host Discovery → Lateral Movement | Automated compromise propagation |
| ARP Host Discovery → ARP Cache Poisoning | Systematic MiTM network preparation |
| SSH Brute Force → Lateral Movement | Post-authentication pivot detected |
| Lateral Movement → Bulk Transfer | Possible staging and exfiltration |
| Lateral Sweep → ICS/HMI Recon | Post-pivot targeting of industrial controls |

The **session summary** and **markdown report** both include a full kill chain activity section for any source that triggered 2+ alerts.

## Allowlist and Asset Configuration

### `config/allowlist.yml`

Mark hosts as authorized scanners. Their alerts are downgraded in severity and confidence but still logged to `siem.log`.

```yaml
authorized_scanners:
  - 192.168.0.100   # Nessus scanner

trusted_ports:
  - port: 443
    name: HTTPS

noisy_protocols:
  - mdns
  - ssdp
  - dhcp
```

### `config/assets.yml`

Map IPs to roles and criticality. Alerts touching critical assets are escalated.

```yaml
assets:
  - ip: 192.168.0.10
    name: DC01
    role: domain_controller
    criticality: critical

  - ip: 192.168.0.20
    name: HMI-01
    role: ics_hmi
    criticality: critical
```

Supported roles: `domain_controller`, `database`, `file_server`, `web_server`, `ics_hmi`, `plc`, `scada`, `router`, `firewall`, `employee_laptop`, `printer`.

## Privacy / Sanitize Mode

Replace all real IPs and MACs in output with anonymous aliases (`HOST_A`, `HOST_B`, ...):

```
python3 main.py --pcap capture.pcap --sanitize
python3 main.py --pcap capture.pcap --sanitize --report results/report.md
```

Output becomes:

```
Threat   : Port Scan: 26 destination ports probed
Source   : HOST_A
Top flow : HOST_A → HOST_B  [TCP, 120 pkts]
Filter   : ip.src == HOST_A && ip.dst == HOST_B
```

Combined with `--report`, the markdown file contains a full IP/MAC mapping table at the bottom (marked with a warning to remove before sharing externally).

## Markdown Report

Export a full analysis report after pcap processing:

```
python3 main.py --pcap capture.pcap --report results/report.md
python3 main.py --pcap capture.pcap --sanitize --report results/report.md
```

The report includes:

- Executive summary table (windows, severity, top source, top detection)
- Per-alert detail with **evidence table** (Signal / Value / Baseline deviation)
- Confidence score and FP risk per alert
- MITRE ATT&CK link per alert
- Possible benign causes per alert
- Recommended analyst actions per alert
- Kill chain activity section (if multi-event sources were observed)
- IP/MAC mapping table (sanitize mode only, marked private)

**Evidence table example:**

| Signal | Value | Baseline |
|--------|-------|----------|
| Unique dest ports | 42 | 8.7x |
| SYN ratio | 0.78 | 5.1x |
| Packet rate | 342 | 3.4x |

## Explain Alert Command

Type `explain alert N` or `next steps alert N` for a full analyst breakdown:

```
question> explain alert 1

=== Alert 1 — [HIGH] Port Scan: 42 destination ports probed ===
Time     : 21:04:20-21:04:30
Severity : HIGH
Confidence: 86%  |  FP Risk: LOW
Flow direction: Internal → Internal
MITRE ATT&CK: T1046  (https://attack.mitre.org/techniques/T1046)

Why this was flagged:
  - 42 destination ports probed — automated service discovery sweep
  - Average 1.3 packets/flow — short-lived connections typical of scanning
  - Deviation: 8.7x above baseline on unique dest ports

Possible benign causes:
  - Authorized vulnerability scanner (Nessus, Tenable, Nmap)
  - Network inventory or asset discovery tool
  - Software update manager checking multiple endpoints
  - Printer or network device discovery (mDNS/SSDP broadcast)

Triage checklist:
  [✓] Source is internal
  [ ] Source is NOT on the authorized-scanner allowlist — verify manually
  [✓] Alert repeated across 4 windows — persistent pattern
  [✓] High port diversity (42 unique dst ports)

Top flow : HOST_A -> HOST_B  [TCP, 120 pkts]
Filter   : ip.src == HOST_A && ip.dst == HOST_B

Recommended analyst actions:
  1. Apply filter: ip.src==HOST_A  to review all ports probed by this host
  2. Verify the source IP is an authorized scanner (Nessus, Nmap, Tenable, etc.)
  3. Check if any probed ports sent a response — a SYN-ACK means a live service is exposed
  ...
```

## Local Network Awareness

The system classifies each alert's flow direction to reduce false positives from broadcast and multicast noise:

| Direction | Meaning |
|-----------|---------|
| `Internal → Internal` | Both source and destination are RFC1918 addresses |
| `Internal → External` | Outbound traffic from an internal host |
| `External → Internal` | Inbound traffic — higher suspicion |
| `Noise` | Multicast / broadcast only — confidence is reduced |

mDNS, SSDP, DHCP, NBNS, LLMNR, and IGMP are pre-classified as noisy protocols in `config/allowlist.yml`.

## Pcap File Mode

Read and analyse a saved `.pcap` or `.pcapng` file:

```
python3 main.py --pcap capture.pcap
```

Apply a Wireshark display filter when reading:

```
python3 main.py --pcap capture.pcap --filter "tcp"
python3 main.py --pcap capture.pcap --filter "ip.addr == 192.168.xxx.xxx"
```

> In pcap mode `--filter` uses **Wireshark display filter** syntax, not BPF.

## Common Options

| Option | Default | Description |
|--------|---------|-------------|
| `--window-seconds N` | 10 | Time window size in seconds |
| `--warmup-windows N` | 6 | Windows used to learn the baseline |
| `--contamination F` | 0.15 | IsolationForest contamination (0.01–0.50) |
| `--display-top-flows N` | 5 | Top-N flows shown per alert |
| `--output-csv PATH` | results/live_traffic_windows.csv | CSV output path |
| `--filter EXPR` | — | BPF filter (live) or Wireshark display filter (pcap) |
| `--tshark-path PATH` | tshark | Path to tshark binary |
| `--max-packets-per-window N` | 50000 | Cap packets kept per window |
| `--no-interactive` | — | Disable question prompt (for automation) |
| `--show-window-events` | — | Show per-window logs in interactive mode |
| `--sanitize` | — | Replace IPs/MACs with HOST_A, HOST_B, ... |
| `--report PATH` | — | Write a markdown analysis report after processing |

## Supported Filter Phrases

| Command | Wireshark Filter |
|---------|-----------------|
| `filter tcp` | `tcp` |
| `filter udp` | `udp` |
| `filter dns` | `dns` |
| `filter mdns` | `mdns` |
| `filter icmp` | `icmp \|\| icmpv6` |
| `filter arp` | `arp` |
| `filter http` | `http` |
| `filter https` | `tls \|\| tcp.port == 443` |
| `filter tls handshake` | `tls.handshake` |
| `filter dhcp` | `dhcp \|\| bootp` |
| `filter syn` | `tcp.flags.syn == 1 && tcp.flags.ack == 0` |
| `filter reset` | `tcp.flags.reset == 1` |
| `filter port 443` | `(tcp.port == 443 \|\| udp.port == 443)` |
| `filter ssh traffic` | `(tcp.port == 22 \|\| udp.port == 22)` |
| `filter rdp traffic` | `(tcp.port == 3389 \|\| udp.port == 3389)` |
| `show traffic from <ip>` | `ip.src == <ip>` (with live match) |
| `show traffic from this ip` | `ip.src == <local IP>` (auto-detect, live mode) |
| `show traffic to <ip>` | `ip.dst == <ip>` (with live match) |
| `show traffic between <ip1> and <ip2>` | `ip.addr == <ip1> && ip.addr == <ip2>` |
